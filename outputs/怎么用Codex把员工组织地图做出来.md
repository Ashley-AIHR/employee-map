# 怎么用 Codex 把员工组织地图做出来

## 这份文档是写什么的

这不是一篇泛泛而谈的 AI 感想。

这是我把一组员工组织地图设计图，真正做成一个可运行前端原型时，实际用 Codex 的过程记录。

重点不是“Codex 能不能写代码”。
重点是：如果你手上有一组设计图，想让它从视觉稿变成一个真的能用、能导入数据、能下钻、能导出的前端系统，Codex 到底应该怎么参与。

## 最终做出来的东西

最后做出来的是一个纯前端的 Employee Map 原型，核心能力包括：

- 浏览器内导入 CSV 花名册
- 自动识别一批常见字段
- 生成组织总览与部门视图
- 点击部门进行下钻
- 计算管理跨度、新人占比、P6+ 占比等指标
- 输出结构信号
- 导出 CSV / JSON / 文本摘要
- 把项目推到 GitHub 供外部验证

GitHub 地址：

`github.com/Ashley-AIHR/employee-map`

## 一开始我没有让 Codex 直接“照着做页面”

这是最关键的一点。

如果你一上来就对 Codex 说：“按这张图做一个页面”，它很容易产出一个看起来很像、但本质上只是静态壳子的东西。

我真正需要的不是“图很像”。
我需要的是：

- 能导入数据
- 能形成员工与部门对象
- 能从部门视角切换
- 能生成风险信号
- 能导出结果

所以第一步不是让 Codex 直接写 HTML，而是先让它理解：
这不是一个宣传页，也不是一张截图复刻题，而是一个“前端产品系统题”。

## 第一步：先让 Codex 理解设计图背后的产品能力

我做的第一个动作，不是让 Codex 写组件，而是让它从设计图里抽象产品结构。

我要它先识别这些模块：

- 上传与字段映射
- 组织总览
- 部门下钻
- 深度分析
- 结构信号
- 报表导出

这一步非常重要。

因为如果 Codex 没有先理解“这套图在讲什么能力”，它后面很容易只会做出：

- 一堆写死的卡片
- 一堆写死的图表
- 一些只能看的假按钮

而不是一个真实可操作的系统。

## 第二步：让 Codex 先搭数据模型，而不是先美化

当我确认产品模块后，下一步是让 Codex 先定义数据结构。

我需要它先有一个统一的员工对象。

比如：

```text
employee_id
name
department
job_title
level
city
contract_type
hire_date
manager
```

同时还要有部门对象，用来承接聚合计算：

- 部门人数
- 管理跨度
- 新人占比
- 高阶人才占比
- 城市分布
- 开放岗位

为什么这一步一定要先做？

因为如果没有统一数据模型，页面上的每一块内容都会变成孤立的假数据。
那就不叫产品，只叫视觉拼贴。

## 第三步：让 Codex 先生成一套真实可跑的样例数据

这一招特别有用。

如果一开始没有真实样例数据，很多页面逻辑根本看不出来对不对。

所以我让 Codex 做的不是写 10 行占位数据，而是生成一整套更像公司的样例数据。

最终它生成的是一套 640 人的模拟公司数据，包含：

- 多个部门
- 每个部门负责人
- 员工 ID
- 岗位名称
- 职级
- 城市
- 合同类型
- 入职时间
- 直属上级

有了这套样例数据以后，前端就可以真的跑出：

- 人数统计
- 组织树节点
- 用工类型分布
- 城市分布
- 司龄结构
- 管理跨度
- 风险信号

这一步的意义是：
让系统一开始就像一个“有内容的产品”，而不是一个只有空状态的展示稿。

## 第四步：让 Codex 把“上传花名册”做成真的

这是我最在意的一步。

很多人让 AI 做系统时，会满足于页面上出现一个“上传”按钮。
但那只是界面语言，不是功能语言。

我要求 Codex 做真的导入流程：

1. 读取本地 CSV 文件
2. 解析内容
3. 识别常见字段别名
4. 映射成统一员工结构
5. 做缺失值、重复 ID、日期格式校验
6. 把结果写入前端状态
7. 让整个页面重新按新数据计算

这一点很重要，因为它决定了系统有没有“输入”。

没有输入能力，再漂亮的总览都只是静态演示。

## 第五步：让 Codex 做“可下钻”，不是“切页面”

很多页面看起来有组织结构，但其实点什么都不会真正联动。

我希望的是：
点击一个部门以后，整个系统上下文都切过去。

所以我让 Codex 做的不是简单跳转，而是部门上下文驱动：

- 当前部门指标变化
- 人员清单变化
- 编制与空缺变化
- 结构信号变化
- 导出内容变化

这一步之后，产品才开始像一个系统，而不是一组视觉稿的切页。

## 第六步：让 Codex 不只给图表，还补“解释层”

这一层也很关键。

图表可以告诉你“发生了什么”，但不一定告诉你“这意味着什么”。

所以我又让 Codex 增加规则判断，把一些结构风险变成可读信号。

比如：

- 管理跨度偏大
- 新人占比偏高
- 高阶人才不足
- 开放 HC 偏多

这样做的结果是，系统不只会展示数字，还能给出一句接近业务解释的话。

这一步能明显提升“产品感”。

## 第七步：让 Codex 把最后一公里补上

真正像产品的东西，不能只停在浏览页面。

所以我还让 Codex 补了几类输出动作：

- 导出当前人员 CSV
- 导出结构信号 CSV
- 导出管理摘要 JSON
- 复制汇报文本

这些动作看起来没有炫酷 UI，但它们会决定这个东西能不能进入真实工作流。

## 第八步：用 Codex 做验证，不只做生成

这也是很多人会忽略的地方。

Codex 的价值不只是“帮我写出来”，还包括“帮我确认它是不是成立”。

这次我会让 Codex做这些验证：

- 检查脚本有没有语法错误
- 确认导入逻辑的字段别名是否覆盖常见表头
- 检查生成数据是不是足够支撑页面
- 确认本地服务能不能跑起来
- 确认 Git 状态和提交结果

也就是说，Codex 在这类任务里，不该只扮演代码生成器，而应该同时扮演：

- 产品拆解助手
- 原型工程师
- 数据脚手架生成器
- 验证助手
- Git 协作助手

## 第九步：把结果推上 GitHub，让外部可以验证

做到这一步，我不想停在“本地跑起来了”。

所以我会继续让 Codex：

- 检查当前改动
- 写合适的提交信息
- 提交到本地仓库
- 推送到 GitHub

这一点对我很重要，因为我希望这个项目不是“我自己说它能用”，而是别人也能去看代码、看结构、看结果。

项目地址是：

`github.com/Ashley-AIHR/employee-map`

## 用 Codex 做这种项目时，真正好用的方式

如果总结成方法，我觉得最有效的方式不是一句话全丢给 Codex，而是按这几个层次推进。

### 1. 先定任务性质

先明确告诉 Codex：

- 这不是单纯 UI 复刻
- 这是纯前端可用原型
- 不要只做静态 HTML
- 页面必须有真实交互和数据流

这一步是在校准方向。

### 2. 再定系统边界

比如明确：

- 只做前端
- 不做后端
- 浏览器内导入 CSV
- 用本地样例数据支撑演示
- 需要可导出

这一步是在避免范围漂移。

### 3. 再让它拆模块

不是直接说“做出来”，而是先让它拆：

- 数据对象
- 视图层
- 导入层
- 校验层
- 聚合层
- 导出层

这一步能让结果稳定很多。

### 4. 每次都要求它验证

不要只让它写完就停。

最好继续让 Codex 做：

- 本地检查
- 语法检查
- 导入测试
- Git 提交

这样它会更像一个完整的开发搭子，而不是一台一次性代码吐出机。

## 哪些地方最容易让 Codex 做偏

这是我这次体感很强的一点。

如果提示不够清楚，Codex 很容易往下面几个方向跑偏：

- 把项目做成静态海报页
- 把设计图当成背景图塞进页面
- 只做漂亮卡片，不做真实数据流
- 做出看起来像产品但其实不能导入的假功能

所以在这类任务里，你要反复强调：

- 我要的是系统，不是图
- 我要的是真实功能，不是假入口
- 我要的是数据驱动，不是写死内容

## 这次我觉得 Codex 真正帮上忙的地方

如果只说“Codex 帮我写代码”，其实说轻了。

这次它真正帮上忙的是：

- 快速把模糊的设计感需求落到结构化模块
- 很快把样例数据骨架搭起来
- 能把导入、状态、导出这些前端逻辑串起来
- 能直接在本地改文件、检查、提交、推 GitHub

换句话说，这次最好用的不是某一段代码，而是它能持续在同一个上下文里推进工作。

## 如果你也想照这个方式用 Codex，可以直接照着这个顺序来

1. 先把设计图和目标讲清楚
2. 先让 Codex 复述“这到底是什么产品”
3. 让它先定义数据模型
4. 让它先造样例数据
5. 再做页面
6. 再做导入
7. 再做校验
8. 再做下钻
9. 再做导出
10. 最后让它验证并推 GitHub

## 最后一句

这次我最大的感受是：

用 Codex 做这种项目，最好的用法不是“让它替你做”，而是“让它跟你一起把一个设计感想法，推进成一个真实系统”。

只要你给它的不是一句模糊的“帮我做个页面”，而是一条清楚的产品落地路径，它能交出来的东西会比单纯的 UI 临摹强很多。
