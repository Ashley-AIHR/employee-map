# 第 10 章：沉淀可复用的 Codex 提示词流程

前九章已经把 `employee-map` 这个案例从 0 到 1 走完整了。最后这一章的任务，不是再重复过程，而是把整套做法沉淀成你以后还能反复复用的流程。

## 本章目标

把这个案例里最有效的 Codex 协作方式，整理成一条可迁移到其他产品案例里的工作流。

## 这个案例最后沉淀出的完整顺序

如果把整个项目压缩成一条教程路径，就是：

1. 明确案例目标与范围
2. 拆出模块与工作流
3. 搭建前端骨架
4. 定义数据模型
5. 生成真实样例数据
6. 实现导入与字段映射
7. 做下钻、信号与导出
8. 完成测试与验证
9. 整理成公开仓库

## 在这个案例里，最值得复用的 8 类提示词

### 1. 定义产品目标

```text
不要只复刻视觉。请把图上的文字也当作产品需求，先定义这个系统的工作流、输入、输出和状态。
```

### 2. 拆模块

```text
先不要写 UI。请拆出核心模块、主要工作流、系统状态和模块之间的依赖关系。
```

### 3. 搭骨架

```text
请先搭一个前端-only 的应用骨架，后续要能接入真实数据、导入、规则和导出。
```

### 4. 定数据模型

```text
请定义员工对象、部门对象，以及这个案例里所有核心页面需要的派生指标。
```

### 5. 造真实样例

```text
请生成一套可重复的 500-800 人样例公司数据，不要用随便凑的占位数据。
```

### 6. 做导入链路

```text
不要只做上传按钮。请实现真实 CSV 导入、字段映射、校验和全局重算。
```

### 7. 做规则与导出

```text
请让当前部门上下文影响所有相关视图，并生成结构信号与导出结果。
```

### 8. 整理公开教程

```text
请把这个案例整理成一个别人第一次进入 repo 就能理解和复现的教程型仓库。
```

## 这一章最重要的结论

这次案例真正有效的，不是某一句神奇 prompt，而是一个稳定顺序：

- 先定目标
- 再拆能力
- 再立数据
- 再做流程
- 再做验证
- 最后公开整理

## 本章完成标准

这一章结束以后，你应该已经不仅拥有一个做完的案例，还拥有一套以后可以继续拿去做别的产品原型的方法。
