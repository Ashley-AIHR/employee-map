# 第 10 章：沉淀可复用的 Codex 提示词流程

前九章已经把 `employee-map` 这个案例从 0 到 1 走了一遍。最后这一章的任务，不是再解释一次项目，而是把这整套做法压缩成你以后能继续复用的教程方法。

## 本章目标

把这次案例里最有效的 Codex 协作方式，提炼成一套可迁移到别的产品项目里的方法。

## 这次案例最值得复用的，不是一句神 prompt

真正有效的，不是某一句神奇提示词，而是这一整套推进顺序：

1. 先定义产品目标  
2. 再拆产品能力  
3. 再锁前端架构  
4. 再定数据模型  
5. 再生成真实样例数据  
6. 再做导入和下钻  
7. 再做校验和测试  
8. 最后整理成公开可学的 repo

这条顺序本身，就是这个案例最有价值的教程结果。

## 你以后最常用到的几类指令

### 1. 先重定义任务对象

`不要只复刻视觉。请把图上的文字也当作产品需求。`

### 2. 先逼出能力表

`先拆出这个系统的工作流、输入、输出和状态，不要先写页面。`

### 3. 先锁实现边界

`这个项目必须是前端-only、JavaScript-only，并且浏览器内完成所有关键流程。`

### 4. 先逼出数据模型

`在打磨 UI 之前，先定义员工、部门和所有派生指标的数据结构。`

### 5. 强制真实规模数据

`先生成一套 500-800 人规模的可重复样例公司数据。`

### 6. 强制导入链路成立

`不要只放上传按钮。请实现真实导入、映射、校验和全局重算。`

### 7. 强制结果可解释

`不要停在图表。请输出结构信号和简短判断。`

### 8. 强制仓库能教会别人

`把这个项目整理成别人第一次进入 repo 就能理解和复现的教程型仓库。`

## 这次案例给出的最重要方法论

如果只留一句最有用的话，那就是：

不要一次性把所有要求扔给 Codex，而是每个阶段都补上一条“让它更真实”的下一步硬约束。

这次 `employee-map` 真正做成，不是因为第一轮就说得完美，而是因为每一轮都在继续逼近真实产品：

- 从图，到产品
- 从页面，到系统
- 从样例，到导入
- 从展示，到判断
- 从本地成果，到公开教程

这一章结束后，你应该已经不仅拥有一个完成的案例，还拥有一套以后可以继续拿去做别的产品原型的方法。
