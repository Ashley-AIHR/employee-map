# 第 09 章：怎么发到 GitHub 并把项目讲清楚

## 为什么“发到 GitHub”不是最后顺手一下

这次用户并不满足于“你本地做好了就行”，而是希望仓库本身也能承担教学作用。

这会让文档要求完全变掉。因为此时 repo 不只是代码容器，它还是：

- 项目介绍页
- 上手入口
- 教程入口
- 可信度证明

## 一个好用的 repo 需要告诉别人什么

像这种学习型前端原型仓库，至少要让第一次进来的人马上知道：

- 这是什么项目
- 这是 learning project，不是生产系统
- 怎么本地跑起来
- 怎么导入测试 CSV
- 主要逻辑在哪个文件
- Codex 是怎么参与构建的

## 为什么要做双语

这次项目天然就有双语场景：

- 参考图和产品语义主要是中文
- GitHub 读者可能有英文用户

如果只写单语，就总会有一边的人需要额外猜测。所以更好的做法，是把最关键的信息两边都讲清楚。

## 这次仓库里实际需要的文档层次

一个相对完整的结构大概包括：

- `README.md`：第一眼介绍和快速上手
- `docs/vibe-coding-with-codex.md`：总目录页
- 分章节双语教程：适合认真看的人
- `outputs/test-employee-map-import.csv`：测试输入

这样 GitHub 页面才会像一个完整的学习项目，而不是只有源码。

## 实际发布顺序怎么更稳

比较顺的顺序通常是：

1. 先保证本地能跑
2. 再验证主流程能用
3. 再准备测试输入
4. 再写 README
5. 再写深度教程
6. 最后一起 push，让仓库整体一致

## 这一章的结论

GitHub 仓库本身就是交付的一部分。代码、文档、测试文件和项目定位，都要一起成立，别人才能真正看懂你做了什么。
