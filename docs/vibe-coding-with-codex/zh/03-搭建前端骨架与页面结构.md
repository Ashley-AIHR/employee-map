# 第 03 章：搭建前端骨架与页面结构

前两章解决的是“做什么”，这一章解决的是“怎么做，而且不要做重”。

## 本章目标

在这个案例里，用最轻的架构把整个产品骨架先立起来，确保它以后能承载真实数据、导入和导出。

## 这次案例选的架构

为了让这个项目能快速落地、方便公开、便于别人本地试跑，我们最后用的是最直接的一套：

- `index.html` 负责挂载页面
- `styles.css` 负责视觉系统和布局
- `script.js` 负责状态、数据、导入、计算、导出和渲染

没有框架，没有后端，没有数据库。

## 为什么这套架构适合这个案例

因为这个项目本身是学习型产品原型，它最需要的是：

- 别人 clone 就能跑
- 本地验证成本低
- 逻辑看得见
- 修改一处就能立刻看到结果

如果一上来就引入复杂框架或服务端，你很容易把重点从“产品实现”切到“基础设施搭建”。

## 在这个案例里，浏览器需要承担什么

因为没有后端，浏览器必须自己做完这些事：

- 承载内置样例数据
- 读取本地 CSV
- 完成字段映射
- 做基础校验
- 计算指标
- 切部门上下文
- 保存状态
- 导出文件

也就是说，这不是静态网页，而是浏览器里的一个小系统。

## 这一章最适合怎么让 Codex 开始干活

在这个阶段，你可以直接让 Codex：

`请先搭一个前端-only 的应用骨架，包含导航、主要视图区域、状态容器和后续可接入真实数据的结构。不要先写死营销文案。`

## 本章产出

这一章结束时，理想状态不是“页面已经华丽”，而是：

- 页面骨架已经有了
- 主要视图已经分区
- 后续数据和交互能接进去
- 没有多余的服务端依赖

架构一旦锁住，后面的每一章都会顺很多。
