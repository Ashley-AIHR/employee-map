# 第 06 章：怎么让 CSV 导入真的可用

如果说上一章让这个项目开始“像一个产品”，那这一章决定的是：它到底是不是一个能被别人拿来试的产品。

上传按钮是最容易作假的地方之一。页面上放一个“导入数据”按钮，并不代表背后真的有导入工作流；很多 demo 其实只是给你看一个按钮，让你脑补它能工作。

但你这次一直把问题往最实操的方向推。你不是只问“像不像”，而是不断问：

- 现在能不能让我自己导入
- 该用什么 CSV
- 给我一份测试文件
- 告诉我怎么用

也正是这些问题，把项目逼进了“不能再靠假上传混过去”的阶段。

在这个项目里，所谓“CSV 导入真的可用”，至少意味着浏览器端要完成八件事：

1. 读取本地文件  
2. 把 CSV 解析成行  
3. 识别常见表头别名  
4. 转成系统内部统一的员工结构  
5. 校验缺失字段和重复 ID  
6. 用新数据替换当前数据集  
7. 触发所有依赖视图重新计算  
8. 把状态存在本地，刷新后还能继续看

做到这里，它才不是一个按钮，而是一条完整工作流。

这一章里特别值得强调的一点，是“字段别名识别”本身就是产品能力，而不只是实现细节。因为现实里的花名册几乎从来不整齐：

- 有的公司写 `employee_id`
- 有的写 `员工编号`
- 有的直接写 `ID`

如果系统只能吃一种最完美的字段名，那它在真实世界里会非常脆弱。  
所以现在 repo 里真正支持的是一整组现实里常见的别名：

- `employee_id / 员工ID / 员工编号 / id / ID`
- `name / 姓名 / 员工姓名`
- `department / department_l2 / 部门 / 二级部门 / 团队`
- `job_title / 岗位名称 / 岗位 / 职位`
- `contract_type / 合同类型 / 用工类型 / 用工形式`
- `hire_date / 入职日期 / 入职时间`

这种设计直接决定“别人拿一个真实文件来试时，会不会第一步就卡死”。

同时，我们还专门在 repo 里放了一份标准测试文件：

- `outputs/test-employee-map-import.csv`

这不是多余动作。因为对外部读者来说，最顺的路径不应该是“先猜系统吃什么格式”，而应该是“先用一份确定成功的样例测通全链路，再换自己的数据”。

一条当前版本能直接导入的 CSV，大致长这样：

```csv
employee_id,name,department,job_title,level,city,contract_type,hire_date,manager
E0001,张三,销售与市场,销售经理,P4,上海,正式员工,2024-03-01,李四
```

也就是说，到这一章结束时，产品才第一次真的具备了“你可以拿文件来试”的资格。

如果你想让 Codex 一开始就别把“上传”做成摆设，一个特别有效的要求是：

`不要只做上传按钮。请实现浏览器内真实 CSV 导入、字段别名映射、基础校验、导入后全局重算，并附带一份标准测试 CSV。`

这一章真正想留下的感受是：  
一个前端原型只要能吃进真实文件，它的可信度就会明显上升。因为从那一刻开始，用户不再只能看你展示什么，而是可以拿自己的输入来试图推翻你。
