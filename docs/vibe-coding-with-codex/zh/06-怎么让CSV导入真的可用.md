# 第 06 章：怎么让 CSV 导入真的可用

## 为什么导入是原型真假的分水岭

上传按钮是最容易做假的地方之一。一个按钮放在页面上，不代表背后真的有工作流。

这次项目之所以开始变得可信，很大一部分就是因为用户不断把问题推进到“你现在能不能让我拿文件测”的层面。

## “真的可用”具体是什么意思

一个前端里的 CSV 导入流程，至少要做到：

1. 在浏览器里读取本地文件
2. 把 CSV 解析成行
3. 识别常见表头别名
4. 转成系统内部统一的员工结构
5. 校验缺失字段和重复 ID
6. 替换当前数据集
7. 让所有页面重新计算
8. 把状态保存在本地

做到这里，它才不是假上传。

## 表头别名为什么是产品能力

现实里的花名册非常不统一：

- 有的叫 `employee_id`
- 有的叫 `员工编号`
- 有的叫 `ID`

如果系统只认一种最完美的字段名，那它在真实世界里就很脆弱。所以“字段别名识别”本身就是产品能力，不只是技术细节。

## 这次 repo 里真实支持的别名

当前实现已经支持一批具体别名：

- `employee_id / 员工ID / 员工编号 / id / ID`
- `name / 姓名 / 员工姓名`
- `department / department_l2 / 部门 / 二级部门 / 团队`
- `job_title / 岗位名称 / 岗位 / 职位`
- `contract_type / 合同类型 / 用工类型 / 用工形式`
- `hire_date / 入职日期 / 入职时间`

## 为什么还要专门准备测试 CSV

因为用户不是开发者视角，他需要一份“立刻能导入成功”的标准输入，来验证整个链路是不是通的。

这也是为什么 repo 里还专门放了测试文件：

- `outputs/test-employee-map-import.csv`

这样别人一进来，不用先猜格式，就能先验证系统工作流。

## 一份好用的 CSV 至少长什么样

像下面这种结构，就很适合当前版本：

```csv
employee_id,name,department,job_title,level,city,contract_type,hire_date,manager
E0001,张三,销售与市场,销售经理,P4,上海,正式员工,2024-03-01,李四
```

现在仓库里也真的带了一份测试文件：

- `outputs/test-employee-map-import.csv`

## 怎么让 Codex 真去做导入

一个很有效的要求是：

`不要只做上传按钮。请实现浏览器内真实 CSV 导入、字段别名映射、基础校验、导入后全局重算，并附带一份标准测试 CSV。`

## 这一章的结论

如果导入是假的，那产品就是假的。原型一旦能吃进真实文件，它的可信度会立刻上一个台阶。
