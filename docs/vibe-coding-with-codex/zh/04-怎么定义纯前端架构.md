# 第 04 章：怎么定义纯前端架构

当第三章把产品能力梳理出来之后，下一步就不该再模糊地讨论“怎么实现”，而是要先把实现边界钉死。  
因为这次项目从一开始就不是一个随便选技术路线的练习，它有非常清楚的约束：

- 只做前端
- 只用 JavaScript
- 不上后端
- 但必须真的可用

这几个条件叠在一起，分量其实很重。因为它意味着很多本来会被默认推给服务端的事情，现在全部要在浏览器里完成。

于是浏览器不再只是负责展示 UI，而要真正承担系统工作：

- 自带样例数据
- 读取本地文件
- 解析 CSV
- 做字段映射
- 做基础校验
- 计算组织指标
- 切换下钻状态
- 存本地状态
- 导出文件

也正因为如此，我们才会反复强调：这不是静态网页，而是浏览器里的一个小系统。

这个约束看起来像是在给自己上难度，但对这个项目来说，反而是非常好的。因为边界一旦足够明确，很多不必要的复杂性就自然被挡在外面了。比如这次我们刻意没有做：

- 登录体系
- API
- 数据库
- 后台任务
- 服务端报表生成

这些东西当然都可以继续往上加，但那会让项目过早变重，掩盖掉这次真正想练的能力：  
如何在没有后端的前提下，仍然把产品工作流在前端做实。

而且这种架构还有一个很实在的好处，就是别人更容易复用：

- clone 下来就能跑
- 启动方式简单
- 不依赖任何外部服务
- 产品逻辑基本都看得见
- 调试成本更低

这和你后面要求把它变成公开学习项目、放到 GitHub 上的目标，恰好是同一个方向。

所以在和 Codex 协作时，这类边界最好说得特别直白，比如：

`所有工作流都必须在浏览器内完成。优先使用本地状态、浏览器文件 API、localStorage 和前端导出，不要引入任何必须依赖服务器的实现。`

这句话看起来像技术约束，其实是在帮整个项目守节奏。只要它在，模型就不太容易中途偷偷长出一个后端假设。

这一章最后真正想留下的判断是：  
架构边界不是形式要求，而是项目的护栏。边界越清楚，Codex 写出来的东西越稳定，后面返工越少。
