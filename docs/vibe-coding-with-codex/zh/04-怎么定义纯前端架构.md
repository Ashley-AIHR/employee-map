# 第 04 章：怎么定义纯前端架构

## 这个约束并不轻

这次项目一开始就有很硬的架构边界：

- 只做前端
- 只用 JavaScript
- 不上后端

这听起来简单，但其实意味着很多原本会交给服务端的工作，都得在浏览器里解决。

## 浏览器要负责什么

因为没有后端，所以浏览器这边必须自己完成：

- 内置样例数据
- 本地文件读取
- CSV 解析
- 字段映射
- 数据校验
- 指标计算
- 条件筛选
- 下钻状态切换
- 本地持久化
- 文件导出

也就是说，这不是“静态网页”，而是一个浏览器内的小系统。

## 为什么这种架构反而适合这个项目

对于学习型项目和原型来说，这种做法很合适，因为它有几个好处：

- 环境启动简单
- 别人 clone 下来就能跑
- 没有后端依赖
- 迭代速度快
- 产品逻辑非常透明

## 哪些东西我们是故意不做的

为了保住学习效率和产品重点，这次刻意没有做：

- 登录体系
- API
- 数据库
- 后台任务
- 服务端报表生成

不是这些不重要，而是它们会过早把项目引向另一条复杂路径，掩盖这次真正想练的能力。

## 怎么跟 Codex 说清楚

这类约束最好说得很直接，比如：

`所有工作流都必须在浏览器内完成。优先使用本地状态、浏览器文件 API、localStorage 和前端导出，不要引入任何必须依赖服务器的实现。`

这样 Codex 就不容易在中途偷偷长出一个后端假设。

## 这一章的结论

边界越清楚，Codex 越稳定。尤其是这种“只做前端但要可用”的项目，越早锁住架构，后面返工越少。
