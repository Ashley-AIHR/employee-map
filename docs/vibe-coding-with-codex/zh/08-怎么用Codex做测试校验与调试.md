# 第 08 章：怎么用 Codex 做测试、校验与调试

## 为什么这一章不能省

很多 AI 项目在“看起来差不多了”的时候就停了，但真正让项目变可靠的，往往是后面这段测试和调试。

这次项目里，用户不断把任务往更实操的方向推，比如：

- 现在页面为什么打不开
- 给我一份能测的 CSV
- 告诉我怎么用
- 哪种 CSV 格式最好用

这些问题会逼着项目从“看着像”变成“真的能跑”。

## 这类项目要测什么

对于纯前端原型，最有价值的检查包括：

- 本地 server 能不能正常返回
- 页面能不能打开
- CSV 能不能被正确解析
- 是否存在重复员工 ID
- 是否缺少必填字段
- 日期格式是否异常
- 导入后指标是否重新计算

这些不是锦上添花，而是决定别人能不能复用你的项目。

## Codex 在调试阶段能帮什么

到了调试阶段，Codex 很适合做这些事：

- 快速读代码结构
- 搜索旧的假文案或写死数据
- 检查 server 状态
- 创建标准测试文件
- 对照现有逻辑找出导入格式要求
- 修补具体错误点

也就是说，它不只是写代码机器，还是一个很好的“现场排障搭子”。

## 为什么可重复样例数据在这里特别重要

如果样例数据是固定的，你的验证结果也会更稳定：

- 更容易确认 bug 是否修复
- 更容易写文档
- 更容易给别人演示

这也是为什么这次不是随手造几条数据，而是用了可重复生成的 640 人样例公司。

## 很实用的调试提示词

比如这些就很有用：

- `找出为什么本地页面拒绝连接，并把预览恢复到可访问状态`
- `按照当前导入逻辑，生成一份一定能导入成功的测试 CSV`
- `搜索源码里所有旧的营销式文案，移除那些和真实产品不相关的内容`

## 这一章的结论

测试是 vibe coding 变成工程的地方。真正的信任感，不是来自页面多炫，而是来自它经得起验证。
