# 第 08 章：怎么用 Codex 做测试、校验与调试

很多 AI 项目都会在一个很危险的时刻停下来：页面看起来已经像了，于是大家默认它“差不多完成了”。  
但这次项目后来之所以慢慢站住脚，不是因为视觉越来越像，而是因为你一直把问题往“可验证”上推。

你问的不是“这张图是不是更像了”，而是：

- 现在为什么打不开
- 给我一份能测的 CSV
- 告诉我怎么用
- 什么格式最好用

这些问题看起来朴素，但它们会把项目逼进一个很健康的阶段：  
不能再靠相似度撑着走，必须真的能被验证。

对于这种纯前端原型，最值得测的东西其实很直接：

- 本地 server 能不能正常返回
- 页面能不能打开
- CSV 能不能被正确解析
- 有没有重复员工 ID
- 有没有缺少必填字段
- 日期格式是不是异常
- 导入之后指标有没有重新计算

这些都不是锦上添花，而是决定别人能不能复用你的项目。

这次 app 里真正落下去的校验也正是这些。当前版本会明确检查：

- 员工记录总数
- 必填字段缺失
- 重复员工 ID
- 日期格式异常

而且导入抽屉本身也会直接告诉用户，浏览器将会：

- 解析文件
- 生成员工清单
- 重建部门结构
- 保存到 `localStorage`

也就是说，系统不仅在做校验，还在告诉用户“它准备怎么工作”。

到了调试阶段，Codex 在这个项目里扮演的角色也发生了变化。它不再只是写代码机器，而更像一个现场排障搭子。它特别适合做这些事：

- 快速读代码结构
- 搜索旧的假文案或写死数据
- 检查 server 状态
- 创建标准测试文件
- 反推当前导入逻辑究竟吃什么格式
- 修补具体坏掉的逻辑

这也是为什么我们后面不只是让它继续写页面，而是让它去做更“工程”的工作：  
确认连接、生成测试 CSV、清理旧文案、验证导入链路、检查数据校验结果。

这里还有一个在调试阶段会越来越明显的好处，就是前面那套 deterministic 的 640 人样例公司。因为一旦样例数据固定，很多事情都会简单下来：

- bug 更容易复现
- 修复前后更容易对比
- 文档更容易写稳定例子
- 演示也更容易保持一致

如果数据每次都在变，你调试的时候会很痛苦，别人看教程时也会困惑。

在这种阶段里，特别好用的提示词通常都很务实，比如：

- `找出为什么本地页面拒绝连接，并把预览恢复到可访问状态`
- `按照当前导入逻辑，生成一份一定能导入成功的测试 CSV`
- `搜索源码里所有旧的营销式文案，移除那些和真实产品不相关的内容`

这一章真正想留下的，不只是“要测试”，而是更深一点的感受：  
测试是 vibe coding 变成工程的地方。真正的信任感，从来不是来自页面多炫，而是来自别人拿到它以后，真的能试，真的能查，真的能验证。
