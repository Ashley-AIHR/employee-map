# 第 05 章：怎么设计数据模型和样例公司

## 为什么先做数据模型

如果没有一套统一的数据模型，页面上的每张图、每个卡片、每个表格都可能各吃各的数据，最后看起来像一体，实际上互相对不上。

所以这次不是先追样式，而是先把“员工对象”定义清楚。

## 最小可用员工模型

这次项目里，至少需要这些字段：

- `employee_id`
- `name`
- `department`
- `job_title`
- `level`
- `city`
- `contract_type`
- `hire_date`
- `manager`

这些字段不是随便选的，而是和后面每个页面要算的东西一一对应。

## 这次 repo 里真实用了什么样的样例公司

`script.js` 里的内置样例公司明确包含：

- 8 个命名部门
- 每个部门自己的负责人
- 每个部门自己的岗位池
- 各部门在 640 人公司中的人数权重
- 10 个城市的分布
- 固定的合同类型规则和入职时间规则

系统默认把这套样例命名为：

- `Ashley_AIHR_640_roster.csv`

## 这些字段能算出什么

有了上面这组字段，就能稳定推导出很多组织分析里最先要看的指标：

- 总人数
- 部门人数
- 新人占比
- 管理跨度
- 高阶人才占比
- 城市分布
- 合同类型分布
- 开放岗位或编制信号

## 为什么样例公司必须足够大

用户特别强调：每一页都要喂像 500 人以上公司的真实数据。

这是个很重要的要求，因为如果你只用十几二十条演示数据，很多结构问题根本暴露不出来：

- 图表会显得很假
- 部门切换没有层次
- 管理跨度没法看
- 风险信号没有意义

后来 repo 里放的是一套可重复生成的 640 人样例公司，这样既有规模感，又便于调试和复现。

## 为什么“可重复”很重要

如果样例数据每次都随机，你会遇到这些麻烦：

- bug 很难复现
- 截图前后对不上
- 导出内容不稳定
- 文档里写的数字下次就变了

而 deterministic 的样例数据可以把这些问题全都压下去。

## 一个很重要但不显眼的细节

这套样例数据还故意加入了业务纹理：

- 职能支持更容易出现外包
- 工程相关团队更容易出现高职级
- 各部门负责人会先插入，便于后面派生管理关系

## 怎么跟 Codex 提这个要求

一个很好用的说法是：

`先生成一套可重复的、500-800 人规模的公司样例数据，字段要能支撑组织分析，不要只给几条占位数据。`

## 这一章的结论

数据模型一旦做得扎实，哪怕没有后端，这个原型也会突然有“像产品”的质感。反过来，数据模型一旦虚，整个系统就很难救。
