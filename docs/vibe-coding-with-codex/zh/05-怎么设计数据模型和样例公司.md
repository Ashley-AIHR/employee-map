# 第 05 章：怎么设计数据模型和样例公司

到了这里，项目终于进入最容易决定真假质感的部分：数据。  
前面关于“可配置、可下钻、可解释”的所有承诺，只要数据层做虚了，立刻就会变成空话。

所以这次我们不是先继续磨 UI，而是先把员工数据模型钉住。最小可用的一组字段是：

- `employee_id`
- `name`
- `department`
- `job_title`
- `level`
- `city`
- `contract_type`
- `hire_date`
- `manager`

这组字段不是随便列出来的。它们和后面每个页面要算的东西都是一一对应的。  
有了它们，系统才能稳定推出：

- 总人数
- 部门人数
- 新人占比
- 管理跨度
- 高阶人才占比
- 城市分布
- 合同类型分布
- 编制与空缺信号

也就是说，这一步不是在给数据表凑字段，而是在给产品铺计算地基。

然后就是你给出的另一个关键要求：  
每一页都要喂像 500 人以上公司的真实数据。

这句话非常重要，因为它一下就把项目从“小 demo 逻辑”推到了“像真实内部工具”的逻辑。  
如果样例数据只有十几条、几十条，很多问题根本不会暴露：

- 图表会很假
- 部门切换没有层次
- 管理跨度没有意义
- 风险信号像在猜
- 导出文件也没有真实感

所以后来 `script.js` 里真正落下去的是一套固定的 640 人样例公司，而不是几条拼凑数据。它明确包含：

- 8 个命名部门
- 每个部门各自的负责人
- 每个部门自己的岗位池
- 不同部门在 640 人公司里的不同人数权重
- 10 个城市的分布
- 固定的合同类型规则
- 固定的入职时间规则
- 可继续派生管理关系的结构

系统默认把这套数据直接命名为：

- `Ashley_AIHR_640_roster.csv`

这里还有一个很重要的选择：这套样例数据不是随机的，而是 deterministic 的。  
这件事看起来像技术细节，但对学习项目尤其重要。因为只要样例数据每次都随机，马上就会出现这些问题：

- bug 难以复现
- 截图前后对不上
- 导出内容不稳定
- 文档里的例子下次就变掉

而固定样例可以让验证、讲解、复盘都稳定下来。

更进一步，我们还故意往这套数据里塞了一点“业务纹理”，让它不要像一个光滑的随机生成器：

- 职能支持更容易出现外包
- 工程相关团队更容易出现高职级
- 各部门负责人会先插入，方便后面生成管理关系

这些小细节会让下钻后的结果更像真的组织，而不是数学练习题。

如果你想把这种要求直接交给 Codex，一个很好用的说法是：

`先生成一套可重复的、500-800 人规模的公司样例数据，字段要能支撑组织分析，不要只给几条占位数据。`

这一章想留下的核心结论是：  
只要数据层足够扎实，哪怕没有后端，这个原型也会开始有产品质感；反过来，如果数据层是空的，页面越精致，假的感觉反而越明显。
