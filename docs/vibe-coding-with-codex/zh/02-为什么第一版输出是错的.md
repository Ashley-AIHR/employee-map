# 第 02 章：为什么第一版输出是错的

既然第一章已经把目标说清楚了，那第二章自然要回答一个问题：为什么第一版还是会走偏，而且偏得几乎是必然的？

答案很简单，因为对模型来说，最显眼的模式从一开始就是“强视觉参考图 + 前端仓库 + 按图做”的组合。只要这个模式没有被及时打断，它就会优先去完成最表面的任务：把页面长得像。

于是第一版最典型的问题就出现了：

- 布局看上去差不多
- 卡片和图表都挺像
- 按钮好像也能点
- 但数字是写死的
- 面板之间没有真实联动
- 上传、下钻、摘要、导出都缺完整工作流

这类结果最大的危险，不是它太差，而是它“看起来已经差不多了”。越像，越容易让人误以为项目已经完成了 70%，实际上真正的产品能力可能只站住了 20%。

这也是为什么你后面的纠偏会那么重要。你不是简单地说“再做好一点”，而是非常明确地在改任务定义：

- 这只是把图放进 HTML
- 要理解图上的字
- 这是一个真实产品，不只是图片
- 不要只有 UI
- 要完整可用的 H5

这几句之所以有力量，不是因为语气重，而是因为它们把问题指到了根上：  
不是实现得不够漂亮，而是理解错了对象。

为什么 Codex 会犯这个错？不是因为它不够聪明，而是因为大模型特别擅长顺着当前最强的模式继续做。只要最强模式还是“视觉复刻”，它就会优先把视觉那层补齐。它不是在拒绝思考，而是在忠实完成它此刻最像的任务。

所以在这种场景下，最有用的纠偏方式不是：

`请再优化一下`

而是：

`你误解了我们在做的东西，它不是页面复刻，而是产品实现`

前一句只会让它继续在错误方向上抛光，后一句才会逼它重建问题空间。

而且这次纠偏绝对不是停留在对话层。你可以直接从仓库后来的变化看出来，它有多实。纠偏之后，多出来的不是几段样式，而是一整层新的交付物：

- 640 人规模的样例公司
- 真正可导入的测试 CSV
- 真实可跑的浏览器工作流
- 双语说明和教程
- 后来又进一步拆成 10 章、双语分文件夹的结构

也就是说，这次纠偏改变的不是页面表皮，而是整个构建方向。

这一章最后想留下的结论是：  
vibe coding 真正的速度，不是第一版有多快，而是第一版一旦方向错了，你能不能立刻承认它错了，并重定义任务。越早纠偏，后面越轻；越晚承认第一版只是个壳，返工就越重。
