# 第 02 章：为什么第一版输出是错的

## 第一版为什么会偏

第一版最典型的问题，就是它更像“把参考图塞进 HTML”，而不是“把产品做出来”。

这很常见。因为当最初请求里，视觉线索比产品约束更强时，模型就容易先完成最表面的那层模式匹配。

于是你会得到：

- 静态页面
- 写死数字
- 假图表
- 好看的按钮
- 但是没有真正的工作流

## 用户的纠偏为什么关键

这次真正改变方向的，不是“请你再优化一下”，而是更本质的那句：

- 你这只是把图放进 HTML
- 我要你理解图上的字
- 这是一个真实产品，不只是图片
- 我不要只有 UI
- 我要完整可用的 H5

这不是在要求“更精细的还原”，而是在纠正“你理解错了任务对象”。

## 为什么 Codex 会犯这个错

不是因为它不行，而是因为大模型非常擅长沿着当前最显眼的模式继续做。

如果最显眼的模式是：

- 一组很强视觉参考图
- 一个前端 repo
- “照着做”的语气

那它就很容易先给你一个“长得像”的答案。

所以问题不在于 AI 会不会写代码，而在于你有没有及时把它拉回正确抽象层。

## 正确的纠偏方式是什么

这种时候，最有用的不是说：

`再做好一点`

而是说：

`你误解了我们在做的东西，它不是页面复刻，而是产品实现`

因为前一句只会让它继续在旧方向上抛光，后一句才会让它重建问题定义。

## 这一章的结论

vibe coding 不是“第一版挺像就算了”，而是“第一版一旦走偏，就立刻改任务定义”。真正的速度，来自早纠偏，不是来自忍受错误方向一直往前写。
