# 第 07 章：怎么做真的下钻和可解释层

到第六章为止，这个项目已经有了真实数据，也有了可用导入。但它还差一件会直接影响“产品感”的事：用户切换视角时，系统到底有没有活起来。

这就是为什么“下钻”和“可解释层”必须放在一起讲。因为很多原型表面上有下钻，实际上只是标题变了；也有很多原型表面上有图表，但始终不给判断。前者让系统不能被深入使用，后者让系统不能被真正理解。

在这个项目里，我们对“下钻”的要求很清楚：  
切到某个部门，不是只切一个高亮状态，而是要让整个页面都换上下文。

也就是说，当你从全公司切到销售与市场，或者切到更具体的组织单元时，下面这些东西都应该一起动：

- 当前人数
- 管理跨度
- 新人占比
- 人员清单
- 职级结构
- 城市分布
- 开放岗位
- 结构信号
- 导出内容

如果这些不联动，那它就只是视觉切换，不是产品里的下钻。

而“可解释”则是另一条同样重要的线。设计图里反复强调“可解释”，本质上是在说：不要把图表当终点。  
图表只是中间层，真正对人有帮助的是一句判断。

系统看到的是数字，但用户真正想读到的可能是：

- 新人占比偏高，说明团队处在扩张期
- 管理跨度偏大，说明中层承压明显
- 高阶人才占比偏低，说明梯队支撑偏薄
- 一线城市集中度过高，说明配置弹性偏弱

也正因为如此，我们后面没有停在“图画出来了”，而是继续做了结构信号规则。

这套可解释层的实现方式，其实并不神秘，核心就是一组明确的前端规则：

- 如果新人占比超过阈值，就提示增长中的承接压力
- 如果管理跨度过大，就提示管理负载偏高
- 如果高阶人才占比过低，就提示梯队支撑不足
- 如果开放 HC 偏多，就提示招聘计划需要跟进

规则不一定复杂，但一定要清楚。因为没有规则，就不会有解释；没有解释，页面再复杂也还停留在展示层。

而一旦把“下钻”和“可解释”两层都做起来，产品的气质就会明显变化。它不再只是“给你看组织情况”的界面，而开始变成一个“帮你读懂组织情况”的工具。

这一章想留下的结论是：  
下钻让系统可以被继续探索，可解释层让系统可以被真正使用。两者加在一起，才开始接近“组织地图”这个名字应该有的分量。
