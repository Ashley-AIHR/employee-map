# 第 10 章：你可以复用的提示词与协作方法

## 为什么不要幻想“一句神 prompt”

这次项目做成，不是因为一开始就有一句万能提示词，而是因为提示词会随着项目阶段不断变。

真正有效的，是一组可以反复复用的“协作模式”。

## 模式 1：先重定义任务对象

`不要只复刻视觉。请把图上的文字也当作产品需求，推断出真实的工作流与交互逻辑。`

这个模式适合用在：模型开始往 mockup 方向跑的时候。

## 模式 2：先锁住架构边界

`这个项目必须是纯前端、只用 JavaScript、能本地直接运行、不能依赖后端。所有工作流都要在浏览器内完成。`

这个模式适合用在：你不想让项目无意中变复杂的时候。

## 模式 3：先逼出数据模型

`在打磨 UI 之前，先定义系统内部的实体、字段、关系和需要计算出来的指标。`

这个模式适合用在：你不想做出一堆漂亮但彼此不一致的图表时。

## 模式 4：要求真实规模样例数据

`先生成一套可重复的、至少 500 人规模的样例公司数据，让每一页都能直接展示真实量级。`

这个模式适合用在：规模感会影响产品真实性的时候。

## 模式 5：要求真实导入链路

`不要只放上传按钮。请实现真实导入流程、附一份可直接测试的标准文件，并告诉我怎么用。`

这个模式适合用在：你想让项目能被别人实际测试的时候。

## 模式 6：要求可解释层

`不要停在图表。请根据规则生成结构信号和简短解释，让结果更像产品判断。`

这个模式适合用在：你希望系统不只是展示信息，而是帮助理解时。

## 模式 7：要求仓库能教会别人

`更新 README 和教程，让第一次进来的人能看懂这是什么、怎么跑、怎么测、Codex 是怎么参与构建的。`

这个模式适合用在：repo 本身也属于交付物的一部分时。

## 一套可直接复用的顺序

你可以直接按这个顺序和 Codex 协作：

1. 先重定义产品，不只看图
2. 锁架构边界
3. 定数据模型
4. 生成真实样例数据
5. 做真实工作流
6. 本地测试
7. 写文档并发布

## 这次我们真实走过的顺序

如果按这次真实协作过程回看，顺序大概是：

1. 从参考图出发
2. 否定静态 HTML 复刻
3. 强化“纯前端但必须可用”
4. 强化“每页都要有 500+ 人规模数据”
5. 强化“要有真实可测的 CSV 导入”
6. 强化“双语文档”
7. 强化“拆成 10 章、分中英文文件夹”

## 这一章的结论

最好的 Codex 协作方式，不是一次性把所有要求都堆进去，而是每次都补上“让这个东西更真实的下一条硬约束”。
