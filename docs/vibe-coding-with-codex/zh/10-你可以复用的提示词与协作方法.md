# 第 10 章：你可以复用的提示词与协作方法

走到最后一章，我们其实已经能回头看清一件事：  
这次项目并不是靠一句神奇 prompt 突然成功的。它真正有效的地方，是你不断在关键节点补上“下一条让它变真实的硬约束”。

也就是说，最值得复用的不是某一句话，而是整套协作节奏。

如果把这次真实过程压缩成一条主线，大概是这样的：

1. 先从参考图出发  
2. 很快否定静态 HTML 复刻  
3. 强化“这是纯前端但必须可用”  
4. 强化“每一页都要有 500+ 人规模数据”  
5. 强化“要有真实可测的 CSV 导入”  
6. 强化“双语文档”  
7. 最后再强化“把经验拆成 10 章，让别人也能学”

这条线比任何单句 prompt 都重要，因为它反映了项目是怎么一层层被逼实的。

如果把这些经验拆成可复用的模式，大致可以沉淀成下面几类。

第一类，是先重定义任务对象：

`不要只复刻视觉。请把图上的文字也当作产品需求，推断出真实的工作流与交互逻辑。`

它适合用在：模型开始沿着 mockup 方向跑的时候。

第二类，是先锁住架构边界：

`这个项目必须是纯前端、只用 JavaScript、能本地直接运行、不能依赖后端。所有工作流都要在浏览器内完成。`

它适合用在：你不想让实现悄悄长出服务端依赖的时候。

第三类，是先逼出数据模型：

`在打磨 UI 之前，先定义系统内部的实体、字段、关系和需要计算出来的指标。`

它适合用在：你不想做出一堆彼此没有共同数据基础的漂亮图表时。

第四类，是强制真实规模数据：

`先生成一套可重复的、至少 500 人规模的样例公司数据，让每一页都能直接展示真实量级。`

它适合用在：项目的可信度和规模感高度依赖数据的时候。

第五类，是强制导入链路成立：

`不要只放上传按钮。请实现真实导入流程、附一份可直接测试的标准文件，并告诉我怎么用。`

它适合用在：你希望别人不是“看你演示”，而是“自己动手试”。

第六类，是强制结果能被解释：

`不要停在图表。请根据规则生成结构信号和简短解释，让结果更像产品判断。`

它适合用在：系统的价值不只是展示，而是帮助理解。

第七类，是强制仓库本身能教会别人：

`更新 README 和教程，让第一次进来的人能看懂这是什么、怎么跑、怎么测、Codex 是怎么参与构建的。`

它适合用在：repo 本身也是交付物的一部分时。

如果把整套协作压成一条你以后可以直接复用的顺序，那大概就是：

1. 先重定义产品，不只看图  
2. 锁住架构边界  
3. 定数据模型  
4. 生成真实样例数据  
5. 做真实工作流  
6. 本地测试  
7. 写文档并发布

这一章最后真正想留下的结论其实很简单：  
最好的 Codex 协作方式，不是把所有要求一次性砸进去，而是在项目的每一个阶段，都补上那条“让它更真实”的下一步硬约束。只要这条主线不断，项目就会越来越像一个真的产品，而不是越来越像一张精致的图。
