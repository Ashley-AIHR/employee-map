# 第 09 章：整理 GitHub 仓库与双语文档

做到这里，案例本身已经能跑了。但一个真正完整的 tutorial，不会停在“本地可用”，还要把它整理成别人进入仓库就能理解和复现的项目。

## 本章目标

把 `employee-map` 整理成一个适合公开学习、公开验证、公开复用的 GitHub 仓库。

## 这个案例的仓库需要承担什么角色

它不只是放代码，还要同时承担：

- 项目介绍页
- 首次上手入口
- 测试说明
- 教程入口
- 可信度证明

## 新读者进入仓库时最应该先看到什么

至少要一眼看懂：

- 这是什么项目
- 它是学习型项目，不是生产系统
- 怎么本地跑起来
- 怎么导入测试 CSV
- 主要逻辑在哪些文件
- 教程从哪里开始看

## 为什么这个案例要做双语

因为它天然横跨两个语境：

- 设计图和业务语义主要是中文
- GitHub 阅读与传播会有英文读者

所以双语不是装饰，而是降低理解门槛。

## 当前仓库里最关键的文档层

在这个案例里，至少需要这些层：

- `README.md`
- `docs/vibe-coding-with-codex.md`
- `docs/vibe-coding-with-codex/zh/`
- `docs/vibe-coding-with-codex/en/`
- `outputs/test-employee-map-import.csv`

## 可以直接给 Codex 的指令

```text
请把这个案例整理成一个别人第一次进入 repo 就能看懂、能运行、能测试、能继续学习的双语学习型仓库。
```

## 本章完成标准

这一章结束以后，仓库应该已经具备：

- 清晰的项目定位
- 可直接上手的说明
- 明确的教程入口
- 方便测试的配套文件

到这里，这个案例才真正完成从“项目”到“教程项目”的转变。
