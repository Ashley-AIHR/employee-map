# 第 09 章：整理 GitHub 仓库与双语文档

系统做完、测试跑通以后，这个案例还差最后一大步：让它从“你本地的成果”变成“别人也能理解、运行、学习的项目”。

## 本章目标

把这个 employee-map 案例整理成一个真正可公开、可复用、可学习的 GitHub 仓库。

## 这次案例里，GitHub 仓库需要承担什么角色

它不只是代码容器，还要同时承担：

- 项目介绍页
- 首次上手入口
- 教程目录
- 测试说明页
- 可信度证明

也就是说，README 和文档本身就是产品体验的一部分。

## 一个新访客进入这个案例时，最应该先知道什么

至少要让他第一眼知道：

- 这是什么项目
- 它是学习型项目，不是生产系统
- 怎么本地跑起来
- 怎么导入测试 CSV
- 主要逻辑在哪个文件
- Codex 在其中做了什么

如果这些信息不清楚，项目再完整，看起来也会发空。

## 为什么这次案例必须双语

因为这个项目天然横跨两个语境：

- 设计图和产品语义主要是中文
- GitHub 传播和教程阅读又会有英文读者

所以最好的做法不是“写完中文再补一点翻译”，而是从入口开始就把最核心的信息两边都讲清楚。

## 这次案例最后沉淀出的文档层次

当前仓库里的结构大致是：

- `README.md`：项目介绍和快速上手
- `docs/vibe-coding-with-codex.md`：教程总入口
- `docs/vibe-coding-with-codex/zh/`：中文教程章节
- `docs/vibe-coding-with-codex/en/`：英文教程章节
- `outputs/test-employee-map-import.csv`：测试导入文件

## 本章怎么让 Codex 执行最好

可以直接要求：

`请把这个项目整理成一个别人第一次进 repo 就能看懂、能运行、能测试、能学习的双语学习型仓库。`

## 本章产出

这一章结束以后，案例就不再只是一个代码仓库，而会变成：

- 一个可演示的产品原型
- 一个可验证的测试项目
- 一份可复用的学习材料
