# 第 06 章：实现 CSV 导入与字段映射

前面几章让系统有了骨架和样例数据，但这还不够。一个真正可用的案例，不能只看内置 demo，必须让别人拿自己的文件试。

## 本章目标

把“导入数据”从一个按钮，做成浏览器里真正能跑通的 CSV 工作流。

## 这次案例里，导入流程至少要完成什么

一个真的前端导入流程，至少要做完这些步骤：

1. 读取本地 CSV 文件  
2. 解析成行  
3. 识别表头  
4. 映射到系统标准字段  
5. 生成员工对象  
6. 做基础校验  
7. 重建部门结构  
8. 刷新所有视图  
9. 保存到本地状态

做到这里，它才叫导入工作流，而不是按钮装饰。

## 为什么表头别名支持是这个案例的关键

真实花名册几乎不会完全统一。  
所以如果导入只认一种字段名，产品几乎立刻就不实用了。

这次案例当前支持的别名包括：

- `employee_id / 员工ID / 员工编号 / id / ID`
- `name / 姓名 / 员工姓名`
- `department / department_l2 / 部门 / 二级部门 / 团队`
- `job_title / 岗位名称 / 岗位 / 职位`
- `contract_type / 合同类型 / 用工类型 / 用工形式`
- `hire_date / 入职日期 / 入职时间`

这就是为什么这一步不是小技术点，而是真正的产品能力。

## 这次案例里给测试者准备了什么

为了让别人最快测通全链路，repo 里放了一份现成可导入的文件：

- `outputs/test-employee-map-import.csv`

它的用途不是为了演示一个文件，而是为了给所有第一次进项目的人一条“先测通，再替换成自己数据”的路径。

## 一条可导入的标准示例

```csv
employee_id,name,department,job_title,level,city,contract_type,hire_date,manager
E0001,张三,销售与市场,销售经理,P4,上海,正式员工,2024-03-01,李四
```
