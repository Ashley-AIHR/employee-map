# 第 03 章：怎么把设计语言翻译成产品能力

确认第一版错在“只复刻了视觉”之后，接下来最关键的动作并不是马上改代码，而是先做一件更像产品经理的事情：把设计图里的语言翻译成产品能力。

因为设计稿天然会把三层东西混在一起：

- 视觉风格
- 页面布局
- 产品语义

如果不主动把这三层拆开，模型几乎一定会优先抓最表面的那一层，也就是视觉。  
而这次项目真正开始变实，就是因为我们不再问“这个区域长什么样”，而开始问“这个区域在产品里到底负责什么”。

这组图后来被翻译出的核心能力，大致就是六类：

1. 花名册上传  
2. 字段映射  
3. 组织总览  
4. 部门下钻  
5. 结构信号  
6. 摘要输出

一旦这张能力表出来，后面的实现路线就不再模糊了。项目从“做页面”真正转入了“做系统”。

举个特别典型的例子。页面上写“字段配置”，如果按界面去理解，它好像只是两列字段摆在那儿；但如果按产品能力去理解，它其实暗示的是一段完整工作流：

- 有原始字段
- 有标准字段
- 有字段映射关系
- 有校验
- 甚至可能有模板复用

也就是说，它不是一张图，而是一个数据入口。

“深度分析”“下钻”也是一样。按页面去看，它像是在切 tab；按产品去看，它真正要求的是：

- 有当前组织范围
- 有选中的部门上下文
- 有基于这个上下文重算的指标
- 有与公司整体的对比
- 有继续往下读的结构判断

一旦这样理解，后面的数据模型、状态管理、导入逻辑、导出逻辑才会自然连在一起。

所以如果你希望 Codex 在这个阶段先别急着画 UI，一个非常有用的要求是：

`先不要急着写代码。先把这些图当成产品需求，列出其中隐含的工作流、状态、输入、输出和衍生指标。`

这句话的价值在于，它会把模型从“页面生成器”切到“产品解释器”。

到这一章结束时，一个合格的中间结果不应该是草图，而应该是一张能力清单。你至少应该开始清楚：

- 需要哪些页面
- 每个页面的输入是什么
- 状态如何切换
- 哪些数据是用户给的
- 哪些数据是系统算出来的
- 哪些结果最终要导出

只有这层先被翻译出来，后面的代码才不会越写越散。

这一章真正想留下的感受是：  
好的设计稿本身已经在说产品，只是它说得不够显式。你要做的，不是让 Codex 盯着图临摹，而是让它先把图里的产品语言翻译出来。翻译得越准，后面的系统就越实。
