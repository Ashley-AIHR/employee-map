# 第 03 章：怎么把设计语言翻译成产品能力

## 为什么要多做这一步

设计稿通常把三层东西混在一起：

- 视觉风格
- 页面布局
- 产品语义

如果不把这三层拆开，Codex 很容易抓住最表面的那层，也就是视觉。

## 这次我们提炼出的产品能力

这组图最后被翻译成了六类核心能力：

1. 花名册上传
2. 字段映射
3. 组织总览
4. 部门下钻
5. 结构信号
6. 摘要输出

一旦这张能力表出来，后面的实现路线就清楚多了。

## 举个具体例子

比如页面写着“字段配置”，这就不该只被理解成“两列字段摆在那儿”。

它真正暗示的是：

- 有原始字段
- 有标准字段
- 有映射关系
- 有校验动作
- 可能还有模板复用

再比如页面写“深度分析”“下钻”，那就不能只是切一个 tab，而应该意味着：

- 有当前组织范围
- 有选中的部门上下文
- 有基于这个上下文重算的指标
- 有和公司整体的对比

## 怎么让 Codex 先做这层思考

很有用的一种提示方式是：

`先不要急着写代码。先把这些图当成产品需求，列出其中隐含的工作流、状态、输入、输出和衍生指标。`

这样 Codex 会先进入“产品解释器”状态，而不是直接进入“页面生成器”状态。

## 你要拿到什么样的结果才算合格

做完这一步，你最好已经有：

- 需要哪些页面
- 每个页面需要什么输入
- 状态如何切换
- 哪些指标是算出来的
- 哪些东西需要导出

到这里，才适合真正开始写界面和逻辑。

## 这一章的结论

设计语言翻译得越准，Codex 就越不容易给你做出“好看但空心”的东西。
