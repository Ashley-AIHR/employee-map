# 第 05 章：生成 640 人样例公司数据

有了数据模型，系统还是空的。  
这一章的任务，是给它喂一套足够像真的公司数据，而且规模必须撑得起整个案例。

## 本章目标

生成一套可重复、可验证、规模足够大、还能体现组织纹理的样例公司数据。

## 为什么这个案例必须用大样本

因为这个产品不是只展示几个数字，它要支持：

- 总览
- 下钻
- 结构信号
- 编制与空缺
- 导入与导出测试

如果样例数据只有几十条，会立刻出现这些问题：

- 图表很假
- 部门切换没有层次
- 管理跨度没有意义
- 结构信号不可信

所以这次最后采用的是 `640` 人样例公司。

## 当前样例公司里包含什么

`script.js` 里的样例数据当前包含：

- `8` 个部门
- `10` 个城市
- 各部门负责人
- 各部门岗位池
- 各部门不同人数权重
- 合同类型规则
- 入职日期规则
- 可派生的管理关系

8 个部门分别是：

- 产品与研发
- 平台工程
- 设计中心
- 销售与市场
- 增长销售
- 大客户销售
- 运营与客户成功
- 职能支持

## 为什么这套样例必须是 deterministic

因为案例教程最怕例子漂移。  
如果每次打开数据都不同，会导致：

- bug 难复现
- 文档例子失效
- 导出结果变来变去
- 演示不稳定

所以这次必须生成“固定逻辑下可重复得到”的样例公司。

## 怎么让这套样例更像真实公司

不是只堆人数，而是要加业务纹理，例如：

- 职能支持更容易出现外包
- 工程团队更容易出现高职级
- 部门负责人优先生成，方便后续建立管理关系

这些细节会直接影响下钻和结构信号的可信度。

## 可以直接给 Codex 的指令

```text
请生成一套可重复的 500-800 人样例公司数据，包含多个部门、负责人、岗位池、职级、城市、合同类型、入职日期和管理关系，不要使用随便凑的占位数据。
```

## 本章完成标准

这一章结束以后，你应该已经拥有：

- 一套足够支撑所有页面的样例公司
- 稳定可复现的数据基础
- 可直接用于截图、测试、导出的案例环境
