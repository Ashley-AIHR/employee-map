# 第 05 章：生成 640 人样例公司数据

数据模型有了，但系统还是空的。要让这个案例真正开始成立，就必须给它一套像样的公司数据，而且规模不能太小。

## 本章目标

为这个项目建立一套可重复、可验证、规模足够大、还能体现组织纹理的样例公司数据。

## 为什么这次案例必须上 500+ 人规模

因为这个产品不是拿来展示几个卡片的，它要支持：

- 组织总览
- 部门切换
- 结构信号
- 编制与空缺
- 导入与导出测试

如果只有几十条数据，很多问题都看不出来：

- 图表会太假
- 部门层次感不明显
- 管理跨度没有意义
- 风险信号像瞎猜

所以这次案例最后落的是一套 `640` 人样例公司。

## 当前 repo 里的真实样例公司结构

`script.js` 里现在内置的是一套固定的 640 人公司，包含：

- `8` 个部门
- `10` 个城市
- 每个部门的负责人
- 每个部门的岗位池
- 各部门不同的人数权重

这 8 个部门分别是：

- 产品与研发
- 平台工程
- 设计中心
- 销售与市场
- 增长销售
- 大客户销售
- 运营与客户成功
- 职能支持

## 为什么这套样例要做成 deterministic

因为教程型项目最怕数据每次都变。  
样例如果不稳定，就会导致：

- bug 不好复现
- 截图对不上
- 导出内容漂移
- 文档里的例子失效

所以这次选择的是可重复生成的固定样例，而不是每次随机一批人。

## 这套样例为什么比“随机造数”更真实

除了人数和部门，这次样例里还故意加入了一点业务纹理：

- 职能支持更容易出现外包
- 工程类团队更容易出现高职级
- 部门负责人会先插入，方便后面生成管理关系

这些细节会让下钻后的结果更像真实组织，而不是数学题。

## 这一章怎么让 Codex 执行最好

很适合直接说：

`请生成一套可重复的 500-800 人公司样例数据，包含多个部门、负责人、岗位池、职级、城市、合同类型、入职日期和管理关系。不要用随便凑的占位数据。`

## 本章产出

这一章结束后，你应该已经有：

- 一套像真的样例公司
- 足够支持所有页面的规模
- 稳定可复现的测试基础

这是整个案例能不能“看着像产品”的关键一步。
