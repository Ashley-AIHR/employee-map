# 第 02 章：拆出核心模块与工作流

这一章开始进入案例的第一步实操：不要急着写页面，而是先把设计图拆成功能表。

## 本章目标

把视觉参考图翻译成一个真正可以开发的产品能力清单，让后面的代码、数据和状态管理都有依据。

## 在这个案例里，我们拆出来的核心能力

这组图最后被翻译成了六类核心能力：

1. 花名册导入  
2. 字段映射  
3. 组织总览  
4. 部门下钻  
5. 结构信号  
6. 报表输出

这六类能力，基本就是整个 `employee-map` 的骨架。

## 怎么从设计图里读出这些能力

这里最重要的不是“看页面长什么样”，而是“看页面在要求什么行为”。

例如：

- “字段配置” 不是两列字段排版，而是字段映射能力  
- “组织总览” 不是几张图，而是需要有汇总指标  
- “深度分析” 不是另一个页面，而是要求支持下钻  
- “报告输出” 不是按钮装饰，而是要有真正的导出对象

## 这一章在代码前应该先产出什么

在案例里，这一章最理想的产出不是 HTML 草图，而是一张能力表。至少要列清楚：

- 每个模块叫什么  
- 它吃什么输入  
- 它输出什么结果  
- 它依赖哪些状态  
- 它和其他模块怎么联动

## 这次案例里最推荐给 Codex 的提示方式

你可以直接这样要求：

`先不要写 UI。请把这组设计图拆成产品能力、主要工作流、系统状态、输入字段和导出结果。`

## 本章产出

走完这一章，你应该手里已经有：

- 一个模块清单
- 一个工作流清单
- 一套后续可以实现的数据和交互骨架

做到这里，Codex 才适合进入实现阶段。
