# 第 02 章：拆出核心模块与工作流

目标和边界清楚以后，下一步不是马上写页面，而是先把整个案例拆成模块和工作流。  
这一步做得越清楚，后面 Codex 写出来的东西越像系统，而不是越像拼图。

## 本章目标

把设计图和产品要求，整理成一个明确的模块表与流程表。

## 这个案例的核心模块

`employee-map` 最后稳定下来的核心模块有 6 类：

1. 数据导入  
2. 字段映射  
3. 组织总览  
4. 部门下钻  
5. 结构信号  
6. 报表输出

这些模块不是“页面装饰区块”，而是整个系统的业务骨架。

## 这个案例的主工作流

如果把它串成一个完整流程，大致是这样：

1. 进入系统，先加载内置样例数据  
2. 用户选择导入自己的 CSV  
3. 系统解析字段并映射到标准模型  
4. 根据员工数据重建组织结构  
5. 用户查看全公司总览  
6. 用户切到某个部门继续下钻  
7. 系统根据规则生成结构信号  
8. 用户导出当前结果

## 每个模块最少要回答什么

拆模块时，不要只写名字。至少要写清楚：

- 输入是什么
- 输出是什么
- 依赖什么状态
- 和其他模块如何联动

例如：

`数据导入`
- 输入：CSV 文件
- 输出：员工数据数组
- 依赖：字段映射规则
- 联动：组织总览、下钻、信号、导出都会刷新

## 为什么这一章必须先做

因为如果不先拆模块，Codex 很容易直接开始填页面，最后导致：

- 状态分散
- 各模块吃不同数据
- 图表之间互相对不上
- 导出和导入没有共享同一套模型

## 可以直接给 Codex 的指令

```text
先不要写 UI。请把这个案例拆成核心模块、主要工作流、系统状态和每个模块的输入输出关系。
```

## 本章完成标准

做完这一章，你应该已经有：

- 一张模块清单
- 一条主工作流
- 模块之间的依赖关系

这一步完成以后，才适合进入页面骨架和代码实现。
