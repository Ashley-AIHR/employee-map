# 第 07 章：做出部门下钻、结构信号与导出

现在系统已经能导入数据了，但它还没真正“活”。要让这个案例从可运行走向可使用，最关键的就是把下钻和可解释层做实。

## 本章目标

让用户切换部门时，不只是看见标题变了，而是让系统的多个模块一起切换上下文，并给出结构判断。

## 在这个案例里，“下钻”到底意味着什么

不是切一个 tab，也不是换一个标题，而是让当前组织上下文影响整个页面。

当用户从“全公司”切到某个部门时，至少这些内容都应该跟着变：

- 当前人数
- 新人占比
- 管理跨度
- 城市分布
- 职级结构
- 人员清单
- 编制与空缺
- 结构信号
- 导出结果

如果这些没有联动，那就只是视觉切换，不是真正的下钻。

## 在这个案例里，“可解释”为什么不能省

因为图表只是中间层，真正对人有帮助的是一句判断。

用户不只是想知道：

- 新人占比是多少
- 管理跨度是多少
- P6+ 占比是多少

用户更想知道：

- 这个部门是不是扩张太快
- 中层是不是带宽过重
- 高阶人才是不是偏薄
- 招聘计划是不是该介入

所以这里必须加一层结构信号。

## 这次案例里结构信号靠什么生成

当前版本主要依赖一组前端规则：

- 管理跨度超过阈值
- 新人占比超过阈值
- 高阶人才占比低于阈值
- 开放 HC 过多

这些规则会把图表数据翻译成判断语言。

## 为什么这一步会让产品感明显上升

因为从这里开始，系统就不只是“展示信息”，而开始“帮助理解信息”。

这也是 employee-map 和一张普通数据大屏之间真正开始拉开差距的地方。

## 本章怎么让 Codex 执行最好

你可以直接要求：

`让部门切换影响所有相关视图，并基于规则生成结构信号。不要只更新标题或局部图表。`

## 本章产出

这一章结束时，你应该已经得到：

- 真的会联动的部门下钻
- 能把数字翻译成人话的结构信号
- 更接近真实产品的分析体验
