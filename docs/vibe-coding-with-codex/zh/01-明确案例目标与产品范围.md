# 第 01 章：明确案例目标与产品范围

这一章不讲抽象方法，直接讲这次 `employee-map` 案例的起点：我们到底要做一个什么产品，以及为什么它不能只停留在“长得像设计图”。

## 本章目标

在真正写代码之前，先把这个案例的目标定义成一个可执行的产品任务，而不是一个模糊的视觉复刻任务。

## 这个案例里，我们最终要做出来的东西

这次项目不是海报页，也不是静态 dashboard，而是一个纯前端、可本地运行、能真的导入花名册并输出组织判断的 Employee Map 原型。

它至少要完成这几件事：

1. 页面一打开就有一套像样的公司样例数据  
2. 用户可以导入自己的 CSV  
3. 导入后页面会根据新数据重算  
4. 用户可以切换不同部门继续下钻  
5. 系统会生成结构信号和摘要  
6. 用户可以导出结果

## 为什么这个定义很关键

如果一开始只把任务定义成“把设计图做出来”，Codex 很容易给出：

- 静态卡片
- 写死数字
- 假图表
- 没有联动的按钮

但这次案例真正要练的，是如何把设计图背后的产品能力做出来。

## 这组设计图在说什么产品

设计图里的关键词，其实已经在定义功能边界：

- “可配置” 对应字段映射、规则配置  
- “可下钻” 对应部门切换和上下文联动  
- “可解释” 对应结构信号和判断语言  
- “组织地图” 对应围绕员工与部门关系展开的分析视图

所以在这个案例里，设计稿不是“长相模板”，而是“功能线索”。

## 这次案例的硬边界

为了让这个项目既轻量又诚实，这次从一开始就锁定了这些边界：

- 前端-only
- JavaScript-only
- 无后端
- 本地可运行
- 但功能必须真实可测

这个边界会直接决定后面所有实现选择。

## 在这个案例里，最值得先告诉 Codex 的一句话

如果你也想从类似设计图出发，第一轮最值得对 Codex 说的话是：

`不要只复刻视觉。请把图上的文字也当作产品需求，先说清楚这个系统的工作流、输入、输出和主要状态。`

## 本章产出

走完这一章后，你应该已经得到一份清晰的产品定义：

- 这是什么系统
- 它不是哪种系统
- 它必须完成哪些核心工作流
- 它有哪些明确边界

没有这一步，后面的代码很容易从第一天起就写偏。
