# 第 01 章：明确案例目标与产品范围

这一章只做一件事：把 `employee-map` 这个案例定义清楚。  
如果这一步模糊，后面无论写多少代码，都很容易掉进“页面像了，但产品没成立”的坑里。

## 本章目标

把案例目标从“复刻设计图”升级成“实现产品原型”。

## 这个案例最后要交付什么

这次要交付的，不是营销页，也不是静态大屏，而是一个纯前端的 Employee Map 原型。它要满足：

- 能本地直接运行
- 自带一套可用样例数据
- 支持本地 CSV 导入
- 支持部门切换和下钻
- 能生成结构信号
- 能导出结果

## 在这个案例里，什么不算完成

下面这些都不算完成：

- 只有好看的页面，没有真实数据
- 图表是写死的，不会根据数据变化
- 有上传按钮，但背后没有导入链路
- 切部门只改标题，不影响其他模块
- 有导出按钮，但没有真实导出对象

## 设计图应该被怎么理解

在这个案例里，设计图不是“长相模板”，而是“功能线索”。

设计图里的这些词，应该直接翻译成产品能力：

- “可配置” -> 字段映射、规则配置
- “可下钻” -> 部门切换、上下文联动
- “可解释” -> 结构信号、判断语言
- “组织地图” -> 以员工和部门关系为中心的分析系统

## 这一章里要锁死的边界

为了让案例聚焦、轻量、可公开，这次直接锁定这些边界：

- 前端-only
- JavaScript-only
- 无后端
- 浏览器内完成关键流程

这意味着后面的导入、计算、状态保存、导出，都必须在前端完成。

## 可以直接给 Codex 的指令

```text
不要只复刻视觉。请把图上的文字也当作产品需求，先定义这个系统的工作流、输入、输出、状态和边界。
```

## 本章完成标准

做完这一章，你应该已经能回答：

- 这个案例到底是什么产品
- 它为什么不是普通 dashboard
- 它必须完成哪些核心工作流
- 技术边界是什么

只有这些说清楚，后面的实现才不会跑偏。
