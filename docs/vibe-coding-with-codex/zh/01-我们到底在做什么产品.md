# 第 01 章：我们到底在做什么产品

如果这套教程只允许有一个真正的起点，那就必须是这一章。因为这次项目一开始最大的问题，不是代码写得不够好，也不是页面长得不够像，而是“我们到底在做什么东西”这件事没有被说透。

最开始，我们手上是一组很成熟的设计参考图。它们视觉很强，信息很满，中文字面上也已经把产品讲得八九不离十。正常情况下，AI 很容易把这种输入理解成一句隐含命令：  
“请把这些图做成网页。”

但你很快就把这条路掐断了。你指出，如果只是把图塞进 HTML，那做出来的只会是一个漂亮的壳，而不是一个能工作的系统。也正是从那一刻开始，这个项目才真正有了清晰的目标。

我们要做的不是一张海报，也不是一组“长得像”的页面。我们要做的是一个纯前端、H5 风格、没有后端但真的可用的员工组织地图原型。它必须能吃进数据、能切组织视角、能算指标、能给解释、能导出结果，而不是只负责发光和排版。

这件事之所以重要，是因为设计图里那些词本来就在暗示产品能力，只是如果只看视觉，很容易忽略它们。比如：

- “可配置” 不是装饰词，它意味着字段映射、规则配置这类能力
- “可下钻” 不是一个标签页，它意味着切部门以后，整个视图都应该换上下文
- “可解释” 不是多写几句文案，它意味着系统必须把图表翻译成判断
- “组织地图” 不是普通 dashboard 的别名，而是在强调组织结构本身才是分析中心

所以真正的阅读方式，不是“把图还原出来”，而是“把图在表达的产品做出来”。

你给出的边界也因此非常关键：

- 只做前端
- 只用 JavaScript
- 不要后端
- 但必须真的可用

这些约束听起来像是在限制发挥，其实是在帮助项目变诚实。因为一旦确认没有后端，所有关键能力都必须在浏览器里站住脚：样例数据、CSV 导入、字段标准化、组织分析、结构信号、结果导出，全都得在前端完成闭环。

所以到这一章的最后，我们终于能把这个产品定义得很具体。所谓“可用原型”，在这次项目里至少意味着：

1. 打开页面时不是空壳，而是直接有一套真实规模的样例公司  
2. 用户可以导入自己的 CSV，而不是只能看固定 demo  
3. 导入之后，所有页面都要按新数据重新计算  
4. 用户可以切到不同部门继续往下钻，而不是永远停留在全公司  
5. 最后还能导出结构信号和摘要，而不是只能看一眼图

你当时那句最关键的纠偏，其实已经把这一章的核心全部点明了：

`this is simply putting pics into html while i want you to understand the words -- it is a fulld esign of a real product not just pics`

它真正做的，不是嫌弃第一版不够精细，而是把任务从“页面复刻”重新定义成了“产品理解与实现”。

这一章最后只想留下一个判断：  
在让 Codex 大量写代码之前，先把“我们到底在做什么产品”说透。很多时候，这一步不是准备工作，而是最核心的工作。
