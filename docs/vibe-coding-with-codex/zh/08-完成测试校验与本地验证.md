# 第 08 章：完成测试、校验与本地验证

功能做完以后，不能靠“看起来差不多”收尾。  
这一章的任务，是把整个案例测到别人也能安心上手。

## 本章目标

让这个前端案例从“能演示”升级成“能验证”。

## 这个案例里最值得检查的项目

对于纯前端产品原型，最关键的检查包括：

- 本地服务能否正常返回
- 页面能否正常打开
- CSV 能否被正确解析
- 是否存在缺失字段
- 是否存在重复员工 ID
- 日期格式是否异常
- 导入后所有核心视图是否同步刷新

## 当前版本已经落下去的校验

当前 app 会明确检查：

- 员工记录总数
- 必填字段缺失
- 重复员工 ID
- 日期格式异常

导入抽屉也会明确告诉用户：

- 浏览器会解析文件
- 生成员工清单
- 重建部门结构
- 保存到 `localStorage`

## 这一章里 Codex 最适合做什么

在测试阶段，Codex 最适合承担这些任务：

- 检查本地服务状态
- 搜索残留写死文案
- 生成标准测试 CSV
- 反查当前导入逻辑支持什么格式
- 验证某次改动有没有引入回归

## 为什么固定样例数据在这里特别重要

因为样例一旦固定，测试结果就稳定：

- bug 更容易复现
- 修复效果更容易比较
- 教程例子不会漂
- 导出内容更容易核对

## 可以直接给 Codex 的指令

```text
请检查这个案例当前的本地可用性、导入校验结果和视图联动情况，并补一份已知可导入成功的测试文件。
```

## 本章完成标准

这一章结束以后，项目应该已经具备：

- 可访问的本地预览
- 可验证的导入链路
- 明确可见的校验反馈
- 可重复的测试环境
