# 第 08 章：完成测试、校验与本地验证

到这一章，系统功能已经基本齐了，但“像能用”和“真的能用”之间，还差最后一段最工程化的工作：测试、校验和调试。

## 本章目标

把这个项目从“看起来不错”推进到“别人真的能打开、能导入、能验证、能发现问题”。

## 在这个案例里，最值得测的是什么

对于一个纯前端原型，最有价值的检查包括：

- 本地 server 能不能正常返回
- 页面能不能正常打开
- 导入文件能不能被正确解析
- 有没有缺少必填字段
- 有没有重复员工 ID
- 日期格式是否异常
- 导入后指标是否真的重新计算

这些检查不是锦上添花，而是这个案例能不能公开给别人试的底线。

## 当前 app 里已经落下去的校验

这次案例里，导入后会明确检查：

- 员工记录总数
- 必填字段缺失
- 重复员工 ID
- 日期格式异常

而且导入抽屉里还会明确告诉用户浏览器接下来会做什么：

- 解析文件
- 生成员工清单
- 重建部门结构
- 保存到 `localStorage`

## 这章里 Codex 最适合做什么

到了这个阶段，Codex 的角色会从“页面生成器”变成“调试搭子”。它特别适合：

- 检查本地服务状态
- 找导入失败的原因
- 生成标准测试文件
- 搜索还残留的假文案
- 查哪些逻辑还在写死
- 修补具体错误点

## 为什么固定样例数据在这里特别有价值

因为一旦样例数据固定，测试结果就稳定：

- bug 更容易复现
- 修复前后更容易比较
- 文档更容易写
- 演示更容易保持一致

这也是为什么前面一定要做 deterministic 的 640 人样例公司。

## 本章怎么让 Codex 执行最好

可以用非常工程化的要求：

- `检查为什么本地页面打不开，并恢复到可访问状态`
- `按当前导入逻辑生成一份一定能导入成功的测试 CSV`
- `搜索源码中所有还残留的静态营销文案并清理`

## 本章产出

这一章结束后，你应该已经拥有：

- 一个本地能正常打开的系统
- 一条可以端到端测试的导入链路
- 明确可见的校验结果
- 一个别人能放心试的前端原型
