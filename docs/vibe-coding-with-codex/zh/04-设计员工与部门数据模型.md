# 第 04 章：设计员工与部门数据模型

页面骨架搭好了，下一步就不能再拖数据模型。  
对于 `employee-map` 这种产品，数据模型就是它的骨头。

## 本章目标

定义一套足够支撑整个案例的员工模型、部门模型和派生指标。

## 员工对象最少需要哪些字段

当前版本最后稳定下来的员工字段是：

- `employee_id`
- `name`
- `department`
- `job_title`
- `level`
- `city`
- `contract_type`
- `hire_date`
- `manager`

## 为什么这组字段刚好够用

因为它们刚好支撑后面所有核心指标：

- 总人数
- 部门人数
- 新人占比
- 管理跨度
- 高阶人才占比
- 城市分布
- 合同类型分布

这就是案例数据模型的标准：字段不是越多越好，而是每个字段都要服务于实际分析。

## 部门模型在这个案例里怎么来

这里没有单独手写一张“部门表”，而是从员工数据里聚合出部门对象。这样能保证：

- 组织总览和人员清单一致
- 下钻后的部门指标一致
- 导出结果和当前视图一致

部门对象会派生出：

- 部门人数
- 部门负责人
- 平均跨度
- 新人占比
- P6+ 占比
- 默认城市
- 预算与开放 HC

## 为什么这一章必须在样式前完成

如果没有统一数据模型，后面最容易发生的事就是：

- 每张图表用不同数据结构
- 清单和指标对不上
- 导入后很难全局刷新
- 导出逻辑必须单独再写一套

## 可以直接给 Codex 的指令

```text
请先定义员工对象、部门对象，以及这个案例里所有核心页面需要的派生指标。不要先美化页面。
```

## 本章完成标准

做完这一章，你应该已经有：

- 一套稳定员工模型
- 一套由员工数据派生出来的部门模型
- 后续总览、下钻、信号、导出的共同基础
